# CodexLight 使用与实现说明

[English](USAGE_AND_IMPLEMENTATION.md) | 简体中文 | [返回文档索引](README.md) | [项目主页](../README.md)

本文合并了项目的操作手册与技术实现说明，适用于固件烧录、手机 AP 配网、有线或无线运行、故障排查和后续代码维护。根目录 [README.md](../README.md) 是项目总入口。

## 硬件连接

| LED | 数据引脚 | 状态 |
| --- | --- | --- |
| 黄色 WS2812B | GPIO5 | 未连接或等待批准 |
| 绿色 WS2812B | GPIO6 | 已连接动画或任务完成 |
| 红色 WS2812B | GPIO7 | Codex 正在工作 |

三颗 WS2812B 分别使用独立 DIN。VCC 接稳定 5 V，GND 与 ESP32-C3 共地，每路 DIN 按原理图串联 330 欧姆电阻。

## 编译与烧录

```powershell
cd C:\path\to\CodexLight\Firmware
pio run -j 1
pio run -t upload --upload-port COM4
```

将 `COM4` 替换为实际端口。烧录后可查看串口输出：

```powershell
pio device monitor --port COM4 --baud 115200
```

正常启动输出类似：

```text
CODEXLIGHT READY
STATUS mode=WIRED active=NONE wifi=DISCONNECTED
```

退出监视器使用 `Ctrl+C`。PlatformIO Monitor 和 Bridge 不能同时占用同一串口。

## 有线连接

在仓库根目录运行：

```powershell
python -m pip install pyserial
python Bridge\codex_light_monitor.py --serial COM4 --baud 115200
```

也可以自动选择常见 ESP32 串口：

```powershell
python Bridge\codex_light_monitor.py --serial auto --baud 115200
```

Bridge 会重复发送 `MODE WIRED`，直到 ESP32 回复 `MODE_OK WIRED`，随后每 2 秒发送当前状态心跳。

## 手机 AP 配网

没有可用 Wi-Fi 凭据时，ESP32 开启：

```text
SSID: CodexLight-XXXX
Password: 123456789
URL: http://192.168.4.1
```

1. 手机连接 `CodexLight-XXXX`。
2. 等待配置页弹出，未弹出时手动访问 `http://192.168.4.1`。
3. 选择电脑也能连接的 2.4 GHz Wi-Fi。
4. 输入 Wi-Fi 密码并保存。
5. ESP32 连接成功后关闭临时 AP，并在后续上电时自动重连。

ESP32-C3 不支持 5 GHz Wi-Fi。重新配网可通过串口发送：

```text
WIFI_CONFIG
CLEAR_WIFI
```

`WIFI_CONFIG` 强制打开配网页；`CLEAR_WIFI` 删除已保存凭据后打开配网页。

## 无线连接

确保电脑与 ESP32 在同一个局域网，然后运行：

```powershell
python Bridge\codex_light_monitor.py --udp --udp-port 4210
```

首次使用纯无线模式前，通过串口发送并保存：

```text
MODE WIRELESS
```

Bridge 先通过 UDP 广播发现设备，再把匹配设备的 MAC 和最近 IP 保存到已被 Git 忽略的 `Bridge/config.local.json`，之后优先使用单播。

## 同时使用有线和无线

```powershell
python Bridge\codex_light_monitor.py --serial COM4 --baud 115200 --udp --udp-port 4210
```

Bridge 会自动发送 `MODE AUTO`。`AUTO` 同时接受两类心跳，有线心跳有效时优先有线；有线失效后可继续使用无线。

## Windows 托盘启动

双击 `Bridge\start_codex_light_tray.bat` 默认以 `AUTO` 模式启动。也可以在 PowerShell 中选择：

```powershell
Bridge\start_codex_light_tray.bat auto
Bridge\start_codex_light_tray.bat wired
Bridge\start_codex_light_tray.bat wireless
```

`wired` 只启用串口。`wireless` 的状态只走 UDP；如果 USB 可用，Bridge 会先通过串口保存 `MODE WIRELESS`，收到确认后立即释放串口，没有 USB 时则使用固件已保存模式。批处理顶部可统一修改串口、波特率和 UDP 端口。运行后可通过托盘右键菜单的 `Connection mode` 直接切换 `Auto`、`Wired only` 和 `Wireless only`，Bridge 会自动重启，托盘状态文字显示当前模式。

## 通信模式与 NVS

```text
MODE WIRED
MODE WIRELESS
MODE AUTO
STATUS
```

模式保存在 Preferences namespace `codexlight` 的 `transport` 键中。普通固件烧录不会清除 NVS。需要完整擦除时：

```powershell
cd Firmware
pio run -t erase --upload-port COM4
pio run -t upload --upload-port COM4
```

## LED 行为

- 未收到有效电脑心跳：GPIO5 黄灯按 1 秒完整周期闪烁。
- 首次建立电脑连接：GPIO6 绿灯闪烁 2 秒。
- Codex 思考、回复或执行工具：GPIO7 红灯常亮。
- 等待批准、权限或用户输入：GPIO5 黄灯常亮。
- `task_complete` 或 `turn_aborted`：GPIO6 绿灯常亮。
- 当前传输超过 6 秒没有心跳：恢复黄灯闪烁。

Bridge 会锁存批准对应的 `YELLOW`，普通并行工具事件和 SQLite 诊断不会覆盖它，直到对应批准调用返回。任务完成后的 `GREEN` 同样会忽略同一轮尾随事件，直到下一次 `task_started` 或 `user_message`。

固件缓存最后一次 LED 帧，并每秒重发一次，用于纠正供电或时序干扰造成的偶发 WS2812B 错误锁存。

## 总体架构

```text
Codex Desktop logs
        |
        v
Bridge/codex_light_monitor.py
        |
        +-- USB CDC serial: GREEN / RED / YELLOW
        +-- LAN UDP: CODEXLIGHT/1 <STATE>
                         |
                         v
                  ESP32-C3 firmware
                         |
                         +-- GPIO5 yellow WS2812B
                         +-- GPIO6 green WS2812B
                         +-- GPIO7 red WS2812B
```

## Bridge 实现

Bridge 读取：

- `~/.codex/sessions/**/*.jsonl`
- `~/.codex/logs_2.sqlite`

它只提取事件类型、工具名和诊断级别来判断状态。发送给 ESP32 的内容仅为颜色状态，不发送 Codex 消息正文、工具输出、API Key 或登录令牌。

| Codex 事件 | 输出 |
| --- | --- |
| `task_started`、消息、推理、工具调用和工具输出 | `RED` |
| 调用需要批准、权限或用户输入 | `YELLOW` |
| `task_complete` 或 `turn_aborted` | `GREEN` |

串口连接后 Bridge 等待 2 秒，再执行模式握手和状态发送。串口与 UDP 默认每 2 秒重发当前状态作为心跳。

## 固件实现

入口文件为 `Firmware/src/main.cpp`，主要模块包括：

- `src/config_portal.cpp`：WiFiManager 非阻塞 AP 配网。
- `src/led.cpp`：三路独立 Adafruit NeoPixel 输出。
- `include/config.h`：GPIO、亮度、端口、超时、AP 和默认模式配置。

每次主循环依次处理串口、WiFiManager、UDP、活动传输选择和 LED 状态。代码没有等待 Wi-Fi 的无限循环，也没有 `while (!Serial)`，因此未联网时仍能使用有线连接。

LED 使用 `NEO_GRB + NEO_KHZ800`，与参考硬件一致。每次状态显示都会更新三路数据并明确关闭非活动灯。

## 常见问题

### 黄灯一直闪

- Bridge 未启动、串口被监视器占用或 COM 端口错误。
- 固件保存为 `WIRELESS`，但当前只启动了有线 Bridge。
- 无线模式下电脑与 ESP32 不在同一局域网。

### 等待批准时黄灯不常亮

- 重启 Bridge，确保正在运行最新的 `codex_light_monitor.py`。
- PowerShell 应显示 `YELLOW approval_needed:<tool>`。
- 托盘模式需要右键托盘图标选择 `Restart monitor`。

### 手机无法连接配置热点

- 确认 AP 密码与 `Firmware/include/config.h` 一致且至少 8 位。
- 暂时关闭手机的自动网络切换或移动数据辅助。
- 手动访问 `http://192.168.4.1`。
- 通过串口发送 `CLEAR_WIFI` 后重试。

### 颜色不正确

当前硬件使用 `NEO_GRB`。如果更换了不同批次灯珠，需要确认 WS2812B 色序、DIN 方向、GPIO 连通性和焊接质量。

## 安全与本地配置

- `Bridge/config.local.json`、`Bridge/logs/` 和 `Firmware/include/wifi_secrets.h` 已被 Git 忽略。
- WiFiManager 调试输出已关闭，固件不会在串口打印 Wi-Fi 密码。
- UDP 协议没有加密和鉴权，只应在可信局域网使用。
- 默认 AP 密码是公开配置，发布实际设备前建议在 `Firmware/include/config.h` 中修改。

## 已移除的旧实现

- 使用 `wifi_secrets.h` 硬编码运行时 Wi-Fi 凭据。
- 自定义阻塞配网页。
- UDP token 配对和 `PAIR_SET`。
- FastLED 多控制器全局刷新。
- 使用普通 `item/completed` 提前判断整轮任务完成。
