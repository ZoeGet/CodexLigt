# CodexLight Implementation Status

中文 | [English](IMPLEMENTATION_STATUS.md)

本文档用于给后续新对话或接手开发者快速了解 CodexLight 当前已经实现的功能、实现方式和关键技术决策。

## 项目目标

CodexLight 是一个基于 ESP32-C3 SuperMini 的 Codex 状态灯。电脑端监听本机 Codex Desktop 日志，判断 Codex 当前状态，并把状态发送给 ESP32。ESP32 控制三颗独立 WS2812B LED 显示状态。

当前状态映射：

| 灯光 | 状态含义 |
| --- | --- |
| 绿灯 | Codex 空闲、任务完成，或一段时间没有新活动 |
| 红灯 | Codex 正在思考、编写、调用工具或执行动作 |
| 黄灯 | Codex 需要批准，或等待明确的用户输入 |

## Codex 日志来源

当前实测可用的主要状态来源是 Codex 会话 JSONL：

```text
C:\Users\<you>\.codex\sessions\YYYY\MM\DD\rollout-*.jsonl
```

每行是一个 JSON 对象。关键事件示例：

```json
{"type":"event_msg","payload":{"type":"task_started","turn_id":"..."}}
```

辅助日志来源是 SQLite：

```text
C:\Users\<you>\.codex\logs_2.sqlite
```

SQLite 只作为完成提示使用，例如诊断日志里出现：

```text
app-server event: item/completed
```

注意：当前 Codex Desktop 本地日志中没有确认稳定的 `task_finished` 或 `waiting_for_user` 单一事件。因此 Bridge 使用组合策略：

- `task_started` 进入红灯。
- agent message/reasoning、工具调用、工具输出期间保持红灯。
- 工具 payload 中出现审批/权限/用户输入标记时进入黄灯。
- app-server 完成提示或静默超时后回到绿灯。

## Bridge 状态判断规则

核心脚本：

```text
Bridge/codex_light_monitor.py
```

主要规则：

| 输入事件 | 输出状态 |
| --- | --- |
| `event_msg.payload.type == "task_started"` | `RED` |
| `agent_message` / `agent_reasoning` | `RED` |
| `response_item.payload.type == "function_call"` | `RED` |
| 工具调用 payload 包含 `require_escalated` / `sandbox_permissions` / `request_user_input` / `approval` / `permission` | `YELLOW` |
| `response_item.payload.type == "function_call_output"` | 通常保持 `RED` |
| SQLite `level == "ERROR"` | `RED` |
| `turn_aborted` | `GREEN` |
| `app-server event: item/completed` 后经过 `--complete-grace` | `GREEN` |
| 超过 `--quiet-timeout` 无新活动 | `GREEN` |

## 连接方式

### 方式一：USB 串口

电脑端命令：

```powershell
python Bridge\codex_light_monitor.py --serial auto --baud 115200
```

串口协议是一行一个 ASCII 状态：

```text
GREEN
RED
YELLOW
```

自动串口扫描依赖 pyserial：

```powershell
pip install pyserial
```

自动模式会优先选择常见 ESP32/USB 串口设备，例如 Espressif、CP210x、CH340/CH341、FTDI、USB Serial、CDC/UART。

### 方式二：无线 UDP

电脑端命令：

```powershell
python Bridge\codex_light_monitor.py --udp --udp-port 4210
```

UDP 默认发送到：

```text
255.255.255.255:4210
```

UDP 协议是一行 ASCII 文本：

```text
CODEXLIGHT/1 GREEN
CODEXLIGHT/1 RED
CODEXLIGHT/1 YELLOW
```

Bridge 默认每 2 秒重复发送当前状态，作为无线心跳。

### 方式三：同时启用 USB 串口和 UDP

电脑端命令：

```powershell
python Bridge\codex_light_monitor.py --serial auto --baud 115200 --udp --udp-port 4210
```

当前 Win10 托盘启动脚本默认使用这个模式。

## Win10 托盘后台运行

入口文件：

```text
Bridge/start_codex_light_tray.bat
```

双击后会启动：

```text
Bridge/CodexLightTray.ps1
```

实现方式：

- 使用 Windows PowerShell 和 .NET `System.Windows.Forms.NotifyIcon` 创建托盘图标。
- 使用隐藏窗口启动 `codex_light_monitor.py`。
- 右键托盘图标可以打开日志目录、重启监听脚本、退出后台程序。
- 不依赖 `pystray` 或其它第三方托盘库。

默认参数在 `start_codex_light_tray.bat` 中：

```bat
set "MONITOR_ARGS=--serial auto --baud 115200 --udp --udp-port 4210"
```

如果只想使用 UDP，可改为：

```bat
set "MONITOR_ARGS=--udp --udp-port 4210"
```

如果只想使用串口，可改为：

```bat
set "MONITOR_ARGS=--serial auto --baud 115200"
```

## ESP32 固件协议

固件入口：

```text
Firmware/src/main.cpp
```

固件同时支持两种输入：

- `Serial`：接收裸状态命令 `GREEN` / `RED` / `YELLOW`。
- UDP：接收带协议前缀的 `CODEXLIGHT/1 GREEN` / `CODEXLIGHT/1 RED` / `CODEXLIGHT/1 YELLOW`。

固件也兼容 UDP 收到裸状态命令，但推荐 Bridge 使用带前缀协议，避免误处理局域网中的其它 UDP 包。

## Wi-Fi 配置

Wi-Fi 凭据不提交到 GitHub。使用方式：

1. 复制示例文件：

```text
Firmware/include/wifi_secrets.example.h
```

2. 改名为：

```text
Firmware/include/wifi_secrets.h
```

3. 填写 Wi-Fi：

```cpp
#define CODEXLIGHT_WIFI_SSID "YourWiFiName"
#define CODEXLIGHT_WIFI_PASSWORD "YourWiFiPassword"
```

`.gitignore` 已忽略：

```text
Firmware/include/wifi_secrets.h
```

没有 `wifi_secrets.h` 时，固件仍然可以编译并通过 USB 串口工作。

## 后续建议

- 给固件增加串口调试输出开关，便于确认 Wi-Fi 和 UDP 收包状态。
- 给 UDP 增加设备发现机制：ESP32 广播 `HELLO`，Bridge 记录 ESP32 IP 后同时 unicast + broadcast。
- 给 Bridge 增加配置文件，避免用户修改 `.bat`。
- 后续如果 Codex Desktop 暴露稳定 `task_finished` / `waiting_for_user` 事件，应优先替换当前的静默超时推断。
